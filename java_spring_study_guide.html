① Java 객체지향 프로그래밍(OOP) 핵심 개념 ▼
🧱 레고 블록으로 이해하기
OOP는 레고 블록 방식으로 프로그램을 만드는 방법입니다.

캡슐화는 레고 블록 안쪽 구조를 숨기고 겉에 있는 버튼만 누르는 것입니다. TV 내부 회로는 몰라도 리모컨 버튼만 누르면 되는 것처럼요.

상속은 기존 설계도를 복사해서 새 기능만 추가하는 것입니다. 일반 자동차 설계도를 받아서 "전기차" 기능만 덧붙이는 것처럼요.

다형성은 같은 명령을 내려도 대상에 따라 다르게 행동하는 것입니다. "소리내!" 라고 하면 개는 "멍멍", 고양이는 "야옹" 하는 것처럼요.

추상화는 공통된 특징만 뽑아서 틀을 만드는 것입니다. "동물"이라는 틀에 "소리 낸다"는 규칙을 정해두는 것처럼요.
📦 캡슐화 — 내부는 감추고, 정해진 방법으로만 접근 가능하게 한다
🧬 상속 — 기존 것을 물려받아 재사용하고 새 기능을 추가한다
🎭 다형성 — 같은 방법으로 호출해도 실제 동작이 다를 수 있다
📋 추상화 — 공통 규칙을 틀로 만들어 강제한다 (interface / abstract)
💡 이 4가지가 Java 코드 전체를 구성하는 핵심 철학입니다

OOP는 캡슐화·상속·다형성·추상화 4대 원칙으로 구성됩니다. SOLID 원칙과 함께 이해해야 합니다.

Java — 다형성 예제
abstract class Animal {
    private String name;
    public Animal(String name) { this.name = name; }
    public abstract String sound();
    public void introduce() {
        System.out.println(name + " says: " + sound());
    }
}
class Dog extends Animal {
    public Dog(String n) { super(n); }
    @Override
    public String sound() { return "Woof!"; }
}
Animal a = new Dog("Rex"); // 다형성 — Animal 타입으로 참조
a.introduce(); // Rex says: Woof!

② Java 컬렉션 프레임워크 (List, Set, Map) ▼
🗄️ 물건 정리함으로 이해하기
데이터를 담는 그릇의 종류가 다릅니다.

List는 번호표가 붙은 줄서기입니다. 1번, 2번, 3번… 순서가 있고, 같은 값이 여러 개 들어가도 됩니다. "대기 순서"처럼요.

Set은 도장 찍기 통입니다. 같은 것은 한 번만 찍힙니다. "이미 방문한 페이지 목록"처럼 중복을 허용하지 않습니다.

Map은 라벨이 붙은 서랍장입니다. "이름"이라는 열쇠로 꺼내면 "홍길동"이 나오고, "점수"라는 열쇠로 꺼내면 "95"가 나옵니다.
📋 ArrayList — 순서 있는 목록. 인덱스로 빠르게 꺼낸다
🚫 HashSet — 중복 없는 목록. "이미 있는지" 확인할 때 유용
🗝️ HashMap — 키-값 쌍. 열쇠(key)로 값(value)을 꺼낸다
⚡ 어떤 그릇을 쓸지 잘 골라야 성능 차이가 크게 납니다
컬렉션구현체특징탐색 복잡도
ListArrayList인덱스 접근, 순서 보장O(1)
ListLinkedList삽입/삭제 빠름O(n)
SetHashSet중복 불가, 순서 없음O(1)
MapHashMapkey-value, 순서 없음O(1)
MapLinkedHashMap삽입 순서 보장O(1)
MapTreeMapkey 기준 정렬O(log n)

③ Java 예외 처리 (try-catch, 사용자 정의 Exception) ▼
🚗 내비게이션 경고로 이해하기
예외(Exception)는 프로그램이 실행 중 "예상치 못한 문제"가 생겼다는 신호입니다.

try 블록은 "이 길로 가봐" 라는 시도입니다. catch 블록은 "만약 막혔으면 이렇게 해" 라는 대비책이고, finally는 "길이 막히든 안 막히든 반드시 해야 하는 것"입니다. (예: 차 시동 끄기)

사용자 정의 Exception은 기본 경고음 외에 "이 상황에만 울리는 특별 경고"를 만드는 것입니다. "사용자를 찾을 수 없을 때"만 울리는 경고처럼요.
🔄 try — 위험할 수 있는 코드를 감싸는 구간
🛡️ catch — 문제가 생겼을 때 실행할 대처 코드
✅ finally — 성공/실패 상관없이 항상 실행 (자원 반납)
⚠️ Checked(반드시 처리) vs Unchecked(런타임 에러) 구분이 중요

Checked Exception(컴파일 강제) vs Unchecked Exception(RuntimeException). Spring에서는 RuntimeException 계열 사용 권장.

Java — 사용자 정의 예외 + try-with-resources
// RuntimeException 상속 — 사용자 정의 예외
public class UserNotFoundException extends RuntimeException {
    private final Long userId;
    public UserNotFoundException(Long id) {
        super("User not found: " + id);
        this.userId = id;
    }
}

// try-with-resources — AutoCloseable 자동 close
try (BufferedReader br = new BufferedReader(new FileReader("f.txt"))) {
    String line;
    while ((line = br.readLine()) != null) System.out.println(line);
} catch (IOException e) {
    log.error("File error", e); // br.close() 자동 호출됨
}
④ Java Stream API 사용법 ▼
🏭 공장 컨베이어 벨트로 이해하기
Stream은 공장 컨베이어 벨트입니다. 물건(데이터)이 벨트 위를 지나가면서 각 작업대를 통과합니다.

filter 작업대 — "불량품 걸러내기" (조건에 안 맞는 것 제외)
map 작업대 — "모양 바꾸기" (데이터를 다른 형태로 변환)
collect 작업대 — "최종 박스에 담기" (결과를 모아서 반환)

핵심은 벨트 위에서 일어나는 일들을 한 줄로 연결할 수 있다는 점입니다. 복잡한 for문 반복을 훨씬 읽기 쉽게 표현합니다.
🔍 filter — 조건에 맞는 것만 남긴다
🔄 map — 각 요소를 다른 형태로 변환한다
📦 collect — 결과를 List, Map 등으로 수집한다
➕ reduce / sum — 모든 요소를 합산한다
💡 중간 연산(lazy)은 최종 연산이 있어야 실제로 실행됩니다

Java 8+ Stream API. 중간 연산(lazy 평가)과 최종 연산(eager 평가)의 구분이 핵심입니다.

Java — Stream 핵심 연산
List<String> names = employees.stream()
    .filter(e -> e.getSalary() > 5000)          // 중간
    .sorted(Comparator.comparing(Employee::getName)) // 중간
    .map(Employee::getName)                        // 중간
    .collect(Collectors.toList());                // 최종

// groupingBy — 부서별 그룹화
Map<String,List<Employee>> byDept = employees.stream()
    .collect(Collectors.groupingBy(Employee::getDepartment));

// flatMap — 중첩 컬렉션 평탄화
List<String> allSkills = employees.stream()
    .flatMap(e -> e.getSkills().stream())
    .distinct()
    .collect(Collectors.toList());
⑤ Java Lambda 표현식 ▼
📝 쪽지 지시사항으로 이해하기
원래는 "이렇게 해줘"라는 지시를 하려면 클래스를 만들고, 메서드를 만들고, 오버라이드하고… 복잡했습니다.

Lambda는 그냥 쪽지 한 장에 지시사항을 적어 건네는 것입니다.

예를 들어 "각 이름을 출력해줘" 라고 하려면:
names.forEach(name -> System.out.println(name))

화살표(->) 왼쪽은 "무엇을 받아서", 오른쪽은 "무엇을 한다"입니다.
✂️ 코드를 짧고 간결하게 표현하는 방법
➡️ (받는 것) -> { 할 일 } 형태로 씁니다
🔗 Stream과 함께 쓰면 매우 강력합니다
📌 @FunctionalInterface — 메서드가 딱 하나인 인터페이스에 사용 가능

Lambda는 함수형 인터페이스(단일 추상 메서드)의 익명 구현체. Java 8+.

Java — Lambda & 함수형 인터페이스 4종
Predicate<String>  isEmpty  = s -> s.isEmpty();       // T → boolean
Function<String,Integer> len = s -> s.length();    // T → R
Consumer<String>  printer  = s -> System.out.println(s); // T → void
Supplier<String>  hello    = () -> "Hello";           // () → T

// 메서드 참조 (:: 연산자)
names.forEach(System.out::println);     // 정적 메서드 참조
names.stream().map(String::toUpperCase); // 인스턴스 메서드 참조
⑥ Java 멀티스레드 기본 개념 ▼
🍳 주방 요리사로 이해하기
단일 스레드는 요리사가 한 명입니다. 국 끓이고, 다 되면 반찬 만들고, 다 되면 밥 차립니다. 순서대로 하나씩이라 느립니다.

멀티스레드는 요리사가 여러 명입니다. 한 명은 국, 한 명은 반찬, 한 명은 밥을 동시에 만들어 빠릅니다.

문제는 재료(공유 자원)를 동시에 잡으려 할 때입니다. 두 요리사가 같은 칼을 동시에 쓰려 하면 충돌이 생깁니다. 이걸 방지하는 것이 synchronized입니다. "이 칼은 내가 쓰는 동안 다른 사람 손대지 마" 표시를 붙이는 것입니다.
🔀 Thread — 프로그램 안에서 동시에 실행되는 작업 단위
🔒 synchronized — 한 번에 하나의 스레드만 접근 허용 (자물쇠)
👁️ volatile — 변수 값을 항상 최신 상태로 읽도록 강제
⚠️ 데드락 — 두 요리사가 서로의 재료를 기다리며 아무도 못 움직이는 상태
🏊 ExecutorService — 요리사(스레드) 팀을 미리 만들어 관리하는 팀장
Java — Thread & ExecutorService
// ExecutorService — 실무 권장 방식
ExecutorService executor = Executors.newFixedThreadPool(4);
executor.submit(() -> processTask());
executor.shutdown();

// synchronized — 임계 영역 보호
public class Counter {
    private int count = 0;
    public synchronized void increment() { count++; }
}

// CompletableFuture — 비동기 처리
CompletableFuture.supplyAsync(() -> fetchData())
    .thenApply(String::toUpperCase)
    .exceptionally(e -> "Error: " + e.getMessage());
데드락 4조건: 상호배제·점유대기·비선점·순환대기 → 하나라도 깨면 방지
volatile — 가시성 보장. 원자성은 보장 안 함

⑦ Spring IoC / DI (제어의 역전 / 의존성 주입) ▼
☕ 카페 바리스타로 이해하기
원래는 커피를 마시려면 내가 직접 원두 갈고, 물 끓이고, 추출했습니다 (new Coffee()).

IoC(제어의 역전)는 "이제 바리스타(Spring)가 알아서 만들어줄게. 넌 그냥 받아 마셔"입니다. 객체를 내가 만드는 게 아니라 Spring이 만들어 줍니다.

DI(의존성 주입)는 바리스타가 만든 커피를 내 손에 쥐어주는 행위 자체입니다. Spring이 필요한 객체를 만들어서 사용할 곳에 직접 넣어줍니다.

덕분에 나는 커피 만드는 법을 몰라도 됩니다. 코드가 서로 덜 엮이게(느슨한 결합) 됩니다.
🏭 Spring 컨테이너가 객체(Bean)를 만들고 관리한다
💉 내가 new 안 해도 Spring이 알아서 넣어준다
🔧 생성자 주입 방식이 가장 안전하고 권장된다
🧩 덕분에 코드 교체·테스트가 쉬워진다

IoC: 객체 생성·라이프사이클을 Spring 컨테이너가 관리. DI는 IoC의 구현 방식. 생성자 주입 권장 — 불변성·순환 의존성 탐지·테스트 용이.

⑧ Spring Boot 프로젝트 구조 이해 ▼
🏢 회사 건물 층별 구조로 이해하기
Spring Boot 프로젝트는 잘 정리된 회사 건물과 같습니다.

controller/ — 1층 안내 데스크. 손님(HTTP 요청)을 맞이하고 적절한 부서로 안내
service/ — 실무 부서. 실제 업무(비즈니스 로직)를 처리
repository/ — 지하 창고(DB). 데이터를 저장하고 꺼내오는 곳
domain/ — 회사에서 다루는 자료의 양식(Entity, DTO)
application.yml — 회사 운영 규칙 (DB 주소, 포트 번호 등 설정)
🚀 Spring Boot = Spring + 자동 설정. 복잡한 XML 설정 없이 바로 실행 가능
📦 내장 Tomcat 포함 — jar 파일 하나로 서버 실행 가능
⚙️ application.yml 파일 하나로 대부분의 설정을 관리

⑨ Spring MVC 동작 원리 (DispatcherServlet) ▼
📮 콜센터 교환원으로 이해하기
웹 요청이 들어오면 DispatcherServlet은 콜센터 교환원입니다. 모든 전화(요청)가 교환원에게 먼저 옵니다.

교환원은 "이 요청은 어느 담당자(Controller)한테 연결해야 하지?" 를 HandlerMapping(전화번호부)에서 찾아봅니다.

담당자가 처리를 마치면 결과를 다시 교환원에게 돌려주고, 교환원이 전화를 건 손님에게 최종 답변을 전달합니다.

핵심은 모든 요청이 한 곳(DispatcherServlet)을 통과한다는 점입니다. 덕분에 공통 처리(보안, 로깅 등)를 한 곳에 집중시킬 수 있습니다.

⑩ Controller / Service / Repository 계층 구조 ▼
🍽️ 식당으로 이해하기
식당 운영을 생각해보면:

Controller (웨이터) — 손님(클라이언트)의 주문을 받아 주방에 전달합니다. 손님이 뭘 원하는지 파악하고 결과를 테이블에 가져다 줍니다.

Service (주방장) — 실제 요리(비즈니스 로직)를 담당합니다. "제육볶음이면 돼지고기+고추장+…" 같은 복잡한 조리 과정을 처리합니다.

Repository (창고지기/식재료 담당) — 냉장고(DB)에서 재료를 꺼내오거나 저장합니다. 주방장이 원하는 재료를 가져다 줍니다.
🏷️ 각 계층은 딱 자신의 역할만 합니다 — 역할 분리
📬 Controller는 직접 DB에 접근하지 않습니다
🔄 데이터는 Entity(DB용) ↔ DTO(API용)으로 변환해서 주고받습니다
Java — 3계층 구조
@RestController @RequestMapping("/api/users") @RequiredArgsConstructor
public class UserController {
    private final UserService userService;
    @GetMapping("/{id}")
    public ResponseEntity<UserResponse> get(@PathVariable Long id) {
        return ResponseEntity.ok(userService.findById(id));
    }
}

@Service @Transactional(readOnly=true) @RequiredArgsConstructor
public class UserService {
    private final UserRepository userRepository;
    public UserResponse findById(Long id) {
        User user = userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
        return UserResponse.from(user); // Entity → DTO
    }
}

⑪ Spring Boot 예외 처리 (@ControllerAdvice) ▼
☎️ 고객센터 불만 처리 창구로 이해하기
서비스에서 문제가 생기면 각 직원(Controller)이 알아서 대응하면 중구난방이 됩니다.

@ControllerAdvice는 중앙 고객센터 불만 처리 창구입니다. 어떤 부서에서 문제가 생기든 한 곳에 모아서 일관된 형태로 고객(클라이언트)에게 안내해줍니다.

"회원을 찾을 수 없어요" → 404 에러 + 친절한 메시지
"입력값이 잘못되었어요" → 400 에러 + 어느 항목이 문제인지 안내
"서버가 터졌어요" → 500 에러 + 상세 내용은 숨기고 일반 메시지만
🎯 전체 앱에서 발생하는 예외를 한 곳에서 처리
📤 클라이언트에게 일관된 형태의 에러 응답 제공
🔒 운영 환경에서는 내부 에러 메시지를 노출하면 안 됨 (보안)
Java — 전역 예외 처리기
@RestControllerAdvice // = @ControllerAdvice + @ResponseBody
public class GlobalExceptionHandler {

    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(UserNotFoundException e) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(ErrorResponse.of("USER_NOT_FOUND", e.getMessage()));
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(MethodArgumentNotValidException e) {
        String msg = e.getBindingResult().getFieldErrors().stream()
            .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
            .collect(Collectors.joining(", "));
        return ResponseEntity.badRequest().body(ErrorResponse.of("VALIDATION_FAILED", msg));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneral(Exception e) {
        // 운영: 내부 메시지 절대 노출 금지
        return ResponseEntity.internalServerError()
            .body(ErrorResponse.of("INTERNAL_ERROR", "서버 오류가 발생했습니다."));
    }
}

📖 페이징도 Pageable로 쉽게 처리

⑮ JPA Entity 설계 ▼
📊 엑셀 시트 설계도로 이해하기
DB 테이블은 엑셀 시트입니다. Entity는 그 시트의 한 행(row)을 Java 객체로 표현한 것입니다.

엑셀에 "users" 시트가 있고 컬럼이 id, name, email이라면, Java에 User 클래스를 만들고 각 컬럼에 해당하는 필드를 만드는 겁니다.

@Entity — "이 클래스는 DB 테이블이야"
@Id — "이 필드가 기본키(primary key)야"
@Column — "이 필드는 DB의 이 컬럼과 연결돼"
🗂️ 하나의 Entity 클래스 = DB의 하나의 테이블
🔑 @Id는 기본키, @GeneratedValue로 자동 증가
🔗 테이블 간 관계는 @ManyToOne, @OneToMany 로 표현
Java — Entity 베스트 프랙티스
@Entity @Table(name="users")
@Getter
@NoArgsConstructor(access=AccessLevel.PROTECTED) // JPA 기본 생성자
public class User {
    @Id @GeneratedValue(strategy=GenerationType.IDENTITY)
    private Long id;

    @Column(nullable=false, length=50)
    private String name;

    @Enumerated(EnumType.STRING) // ORDINAL 사용 금지
    private UserRole role;

    @ManyToOne(fetch=FetchType.LAZY) // LAZY 권장
    @JoinColumn(name="team_id")
    private Team team;

    @CreatedDate @Column(updatable=false)
    private LocalDateTime createdAt;
}
⑯ JPA 영속성 컨텍스트 ▼
🗒️ 메모장 + 자동 저장으로 이해하기
JPA는 DB에서 데이터를 꺼내오면 즉시 버리지 않고 메모장(1차 캐시)에 잠깐 올려두고 관리합니다.

같은 id로 다시 조회하면 DB에 다시 가지 않고 메모장에서 바로 꺼내줍니다 (빠름).

그리고 수정을 해도 save()를 따로 안 불러도 됩니다. 메모장에 있는 동안 변화를 감지해서 트랜잭션이 끝날 때 자동으로 DB에 반영합니다. 이것이 "변경 감지(Dirty Checking)"입니다.
💾 1차 캐시 — 같은 트랜잭션 안에서 같은 데이터는 DB 재조회 없이 반환
🔍 변경 감지 — 엔티티 수정 후 save() 없어도 트랜잭션 끝에 자동 UPDATE
⏱️ 쓰기 지연 — INSERT/UPDATE를 모아뒀다가 한 번에 실행
⚠️ 트랜잭션 밖에서는 영속성 컨텍스트가 없어 Lazy 로딩 불가
엔티티 4가지 상태
비영속(Transient) — new 생성, 컨텍스트와 무관
영속(Managed) — em.persist() / 조회. 변경 감지 대상
준영속(Detached) — em.detach() / 컨텍스트 종료. 변경 감지 안 됨
삭제(Removed) — em.remove(). flush 시 DELETE
핵심 기능
1차 캐시 — 동일 트랜잭션 내 같은 ID 재조회 시 DB hit 없음
동일성 보장 — 같은 ID 엔티티 == 비교 true
Dirty Checking — flush 시점에 스냅샷과 비교, 변경분만 UPDATE
⑰ JPA Fetch 전략 (Lazy / Eager) ▼
🍱 배달 vs 즉석 조리로 이해하기
User를 조회할 때 관련된 Team 정보를 언제 가져올지의 전략입니다.

EAGER(즉시 로딩) — 음식 주문하면서 바로 모든 사이드 메뉴까지 다 시키는 것. User 조회할 때 Team도 무조건 같이 가져옵니다. 쓸 일이 없어도 항상 가져와서 낭비가 생깁니다.

LAZY(지연 로딩) — 일단 메인만 시키고, 사이드 메뉴는 먹고 싶을 때 추가 주문. User만 먼저 가져오고, Team이 실제 필요한 순간에 추가로 조회합니다. 필요할 때만 가져오므로 효율적입니다.
✅ LAZY 권장 — 필요한 시점에만 DB 조회. 성능 절약
⚠️ EAGER는 예상치 못한 추가 쿼리와 N+1 문제를 유발
🔗 LAZY + FETCH JOIN으로 필요할 때만 한 번에 가져오는 것이 실무 패턴
전략로딩 시점기본값권장
LAZY실제 접근 시점@OneToMany, @ManyToMany✅ 권장
EAGER엔티티 로딩 즉시@ManyToOne, @OneToOne⚠️ 지양

⑱ JPA N+1 문제 ▼
🛒 편의점 심부름으로 이해하기
학생 100명의 소속 반 정보를 가져오려 할 때의 비효율을 생각해보세요.

N+1 문제:
1. "학생 목록 가져와" (1번 조회)
2. "1번 학생 반 정보 가져와" (1번 조회)
3. "2번 학생 반 정보 가져와" (1번 조회)
… 100번 반복 = 총 101번 DB 조회 😱

해결책: "학생이랑 반 정보 같이 한 번에 가져와" (JOIN FETCH) = 총 1번 DB 조회 ✅

이것이 N+1 문제입니다. 1번 조회해서 N개의 결과가 나오면, 각각에 대해 N번 추가 조회가 발생하는 현상.
💣 1번 조회 결과에서 N번 추가 조회가 발생하는 성능 문제
🔧 해결책 1: FETCH JOIN — 한 번의 쿼리로 연관 데이터 같이 조회
🔧 해결책 2: @EntityGraph — FETCH JOIN보다 간편한 선언 방식
🔧 해결책 3: @BatchSize — IN 절로 묶어서 쿼리 수 감소
Java — N+1 발생 & 해결
// ❌ N+1: users(1) + team 조회(N번)
users.forEach(u -> System.out.println(u.getTeam().getName()));

// ✅ 해결 1: FETCH JOIN
@Query("SELECT DISTINCT u FROM User u JOIN FETCH u.team")
List<User> findAllWithTeam();

// ✅ 해결 2: @EntityGraph
@EntityGraph(attributePaths={"team"})
List<User> findAll();

// ✅ 해결 3: @BatchSize (컬렉션)
@BatchSize(size=100)
@OneToMany(mappedBy="user", fetch=FetchType.LAZY)
private List<Order> orders;
⑲ SQL 기본 문법 (SELECT, INSERT, UPDATE, DELETE) ▼
📚 도서관 사서로 이해하기
DB는 거대한 도서관입니다. SQL은 사서에게 내리는 명령어입니다.

SELECT — "이 조건에 맞는 책 찾아줘" (조회)
INSERT — "새 책을 이 위치에 꽂아줘" (추가)
UPDATE — "이 책의 제목을 바꿔줘" (수정)
DELETE — "이 책 버려줘" (삭제)

⚠️ DELETE는 WHERE 없이 쓰면 모든 책이 사라집니다. 운영 DB에서 실수하면 큰일 납니다!
🔍 SELECT — 데이터 조회. WHERE로 조건, ORDER BY로 정렬
➕ INSERT — 새 행(row) 추가
✏️ UPDATE — 기존 데이터 수정. WHERE 필수!
🗑️ DELETE — 행 삭제. WHERE 필수! 없으면 전체 삭제

WHERE department = '개발팀' AND salary > 3000000
ORDER BY salary DESC LIMIT 10 OFFSET 0;

INSERT INTO users (name, email, created_at)
VALUES ('홍길동', 'hong@example.com', NOW());

⑳ SQL JOIN (INNER, LEFT, RIGHT) ▼
👥 두 명부 합치기로 이해하기
학생 명부(A)와 도서 대출 명부(B)를 합쳐서 "누가 무슨 책을 빌렸나" 보고 싶다면:

INNER JOIN — 양쪽 명부에 모두 있는 학생만 보여줌 (책을 빌린 학생만)
LEFT JOIN — 학생 명부(A) 기준. 책을 안 빌린 학생도 포함, 대출 정보 없으면 빈칸
RIGHT JOIN — 대출 명부(B) 기준. 퇴학한 학생이 빌린 책도 포함

실무에서는 LEFT JOIN을 가장 많이 씁니다.
🔵 INNER JOIN — 두 테이블 모두에 일치하는 행만
⬅️ LEFT JOIN — 왼쪽 테이블 전체 + 오른쪽 일치하는 행 (불일치는 NULL)
⚠️ NULL 값이 섞이므로 IS NULL 체크를 잊지 말 것
JOIN결과
INNER JOIN양쪽 모두 일치하는 행
LEFT JOIN왼쪽 전체 + 오른쪽 일치 (불일치=NULL)
RIGHT JOIN오른쪽 전체 + 왼쪽 일치 (불일치=NULL)
SQL — LEFT JOIN
SELECT c.name, COUNT(o.id) AS order_count
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
GROUP BY c.id, c.name
ORDER BY order_count DESC;
㉑ SQL GROUP BY / HAVING ▼
📊 반별 평균 성적 계산으로 이해하기
전체 학생 성적표가 있을 때 "반별 평균 점수"를 구하고 싶다면:

GROUP BY 반 — 1반끼리, 2반끼리, 3반끼리 묶기
AVG(점수) — 각 그룹의 평균 계산
HAVING AVG(점수) >= 80 — 그 중에서 평균 80점 이상인 반만 보여줘

WHERE vs HAVING 차이:
WHERE는 "묶기 전에" 걸러냄 (전학생 제외 후 집계)
HAVING은 "묶은 후에" 걸러냄 (집계 결과에 조건)
🗂️ GROUP BY — 같은 값끼리 묶어서 집계 (COUNT, SUM, AVG, MAX, MIN)
🔍 HAVING — GROUP BY 후 집계 결과에 조건 적용
💡 WHERE는 집계 전 필터, HAVING은 집계 후 필터
SQL — GROUP BY / HAVING
SELECT department, COUNT(*) AS cnt, AVG(salary) AS avg_sal
FROM employees
WHERE active = 1           -- 그룹화 전 조건
GROUP BY department
HAVING COUNT(*) >= 5       -- 그룹화 후 조건 (집계함수 사용 가능)
ORDER BY avg_sal DESC;
㉒ SQL 서브쿼리 ▼
❓ 질문 속의 질문으로 이해하기
"평균 점수보다 높은 학생을 보여줘"라는 요청을 생각해보세요.

이걸 처리하려면 먼저 "평균 점수가 얼마야?"를 물어봐야 합니다. 그 답을 가지고 "그것보다 높은 학생"을 찾습니다.

서브쿼리는 이처럼 쿼리(질문) 안에 또 다른 쿼리(질문)를 넣는 것입니다. 괄호 안에 작은 SELECT를 넣어 그 결과를 바깥 쿼리에서 사용합니다.
🔲 괄호() 안에 또 다른 SELECT문을 넣는 구조
📌 WHERE 안에서 동적인 비교값을 구할 때 유용
💡 IN, EXISTS로 목록 포함 여부도 확인 가능
SQL — 서브쿼리 패턴 4종
-- ① 스칼라 서브쿼리
SELECT * FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

-- ② IN 서브쿼리
SELECT name FROM employees
WHERE department_id IN (
    SELECT id FROM departments WHERE location='서울');

-- ③ EXISTS
SELECT * FROM customers c
WHERE EXISTS (
    SELECT 1 FROM orders o WHERE o.customer_id=c.id);

-- ④ FROM 인라인 뷰
SELECT dept, avg_sal FROM (
    SELECT department AS dept, AVG(salary) AS avg_sal
    FROM employees GROUP BY department) sub
WHERE avg_sal > 4000000;
㉓ SQL 인덱스(Index) 기본 개념 ▼
📖 책의 목차/색인으로 이해하기
500페이지 책에서 "영속성 컨텍스트"를 찾을 때, 책을 처음부터 한 장씩 넘기는 것(풀스캔)과 뒤 색인(Index)에서 바로 쪽수를 찾는 것의 차이입니다.

DB 인덱스는 특정 컬럼의 색인 파일을 따로 만들어 두는 것입니다. "email" 컬럼에 인덱스를 만들면 이메일로 검색할 때 전체 테이블을 안 보고 색인만 보면 됩니다.

⚠️ 단점도 있습니다. 색인 파일을 따로 유지해야 하므로, 데이터 추가/수정/삭제 시 색인도 같이 업데이트해야 해서 쓰기가 느려집니다.
⚡ 인덱스 있으면 검색이 엄청 빨라짐 (전체 조회 → 인덱스 조회)
⚠️ INSERT/UPDATE/DELETE는 느려짐 (인덱스도 같이 갱신)
🎯 중복이 적은 컬럼(이메일, 주민번호)에 효과적
🚫 함수 적용(YEAR(date)), 앞 와일드카드('%홍%')는 인덱스 무효화
SQL — 인덱스 생성 & 주의사항
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_orders_user_date ON orders(user_id, created_at);

// ❌ 인덱스 무효화 (Full Scan 발생)
WHERE YEAR(created_at) = 2024   -- 함수
WHERE name LIKE '%홍%'            -- 앞 와일드카드

// ✅ 인덱스 활용
WHERE created_at >= '2024-01-01'  -- 범위 조건
WHERE name LIKE '홍%'              -- 접두사 검색
클러스터드 인덱스: PK 기준 물리 정렬 (테이블당 1개)
복합 인덱스: 선두 컬럼부터 적용됨. 순서 중요

㉙ HTTPS / SSL 보안 프로토콜 ▼
🔒 엽서 vs 자물쇠 상자로 이해하기
HTTP는 엽서를 보내는 것과 같습니다. 누구나 길거리에서 그 내용을 훔쳐볼 수 있죠. (비밀번호가 그대로 노출됨)

HTTPS는 데이터를 금고에 넣고 자물쇠를 채워 보내는 것입니다. 중간에 누가 가로채도 열쇠가 없으면 안을 볼 수 없습니다.

SSL/TLS는 그 금고와 자물쇠를 만드는 보안 기술의 이름입니다. 브라우저 주소창의 초록색 자물쇠 아이콘이 바로 HTTPS를 쓰고 있다는 증거입니다.
🛡️ 데이터를 암호화하여 해킹(스니핑) 방지
🆔 이 사이트가 "진짜"인지 신뢰할 수 있는 기관(CA)이 보증
🔐 비밀번호, 카드번호 입력 시 필수 중의 필수

HTTPS(HTTP over SSL/TLS)는 대칭키 암호화와 비대칭키(공개키) 암호화를 혼합하여 사용합니다.

SSL Handshake 3단계
1. 탐색(Hello) — 브라우저가 서버에 접속, 암호화 방식 협의
2. 인증(Certificate) — 서버가 인증서를 전달, 브라우저는 신뢰 기관(CA) 확인 후 공개키 획득
3. 키 교환(Key Exchange) — 비대칭키로 안전하게 대칭키를 전달하여 이후 통신 암호화
암호화 방식특징용도
대칭키암호화/복호화 키 동일데이터 본문 전송 (빠름)
비대칭키공개키/개인키 쌍 사용대칭키를 안전하게 전달할 때

기본 포트: 443 (HTTP는 80)

🏢 건물 수위/관리인으로 이해하기
Tomcat은 Java 웹 애플리케이션을 실행하기 위한 서버 프로그램입니다.

비유하자면 건물 관리인과 같습니다. 외부에서 방문객(HTTP 요청)이 오면 관리인이 맞아서 "몇 호실 가세요?"(URL 확인) 하고 적절한 사무실(Servlet/Controller)로 안내합니다.

Spring Boot는 Tomcat을 내장해서 별도 설치 없이 java -jar app.jar 한 줄로 서버를 바로 실행할 수 있습니다.
🚪 기본 포트 8080에서 HTTP 요청 대기
🏊 스레드 풀로 동시 요청 처리 (기본 200개)
📦 Spring Boot = 내장 Tomcat 포함. 별도 설치 불필요
📤 전통적 배포: WAR 파일 → 외장 Tomcat webapps 폴더에 배치

㉜ Apache 웹 서버와 WAS 차이 ▼
🏬 편의점 안내데스크 vs 뒷주방으로 이해하기
웹 서버 (Apache/Nginx) — 편의점 안내 데스크입니다. 사전에 준비된 물건(정적 파일: 이미지, CSS, HTML)을 그냥 꺼내줍니다. 빠르고 단순합니다.

WAS (Tomcat) — 뒷주방입니다. "커피 한 잔 만들어줘"처럼 요청에 따라 동적으로 작업(Java 코드 실행, DB 조회)을 수행합니다.

실무에서는 Nginx가 앞에서 정적 파일 처리 + SSL + 부하분산을 담당하고, 뒤에 Tomcat이 비즈니스 로직을 처리하는 구조로 같이 씁니다.
🖼️ 웹 서버 — 이미지, CSS, HTML 같은 정적 파일 제공 특화
⚙️ WAS — Java 코드 실행, DB 조회 등 동적 처리
🔗 보통 Nginx(웹서버) → Tomcat(WAS) → DB 구조로 구성
구분Web ServerWAS
처리정적 콘텐츠동적 콘텐츠 (Java)
예시Apache httpd, NginxTomcat, JBoss, WebLogic
역할리버스 프록시, SSL 처리비즈니스 로직 실행
실무 아키텍처
Client → Nginx (SSL, 정적파일, L7 LB) → Tomcat → DB
㉝ 전자정부프레임워크 구조 이해 ▼
📐 공공기관 표준 업무 매뉴얼로 이해하기
공공기관은 업무를 저마다 다르게 처리하면 혼란스럽습니다. 그래서 정부가 "이 방식으로 만들어라"는 표준 매뉴얼(전자정부프레임워크)을 만들었습니다.

Spring 기반으로 만들어진 공공부문 표준입니다. SI 공공 프로젝트에 참여하면 이 프레임워크를 씁니다.

일반 Spring Boot와 가장 큰 차이는 JPA 대신 MyBatis를 씁니다. MyBatis는 SQL을 XML 파일에 직접 작성하는 방식입니다.
🏛️ 공공기관 SI 프로젝트의 사실상 표준 프레임워크
🗄️ JPA 대신 MyBatis 사용 — SQL을 XML에 직접 작성
📦 배포: WAR 파일 → 외장 Tomcat (Spring Boot 내장 Tomcat 방식이 아님)
🔒 공통 컴포넌트 제공: 파일 업로드, 엑셀, 페이징, 암호화 등
기반: Spring Framework 4.x + MyBatis (SQL Mapper)
MVC: Spring MVC + Tiles (레이아웃)
보안: Spring Security 기반, ARIA 암호화 지원
배포: 외장 Tomcat + WAR 방식
XML — MyBatis Mapper 구조
<mapper namespace="com.example.mapper.UserMapper">
  <select id="selectUserList" resultType="UserVO">

    <where>
      <if test="userName != null">
        AND USER_NM LIKE '%' || #{userName} || '%'
      </if>
    </where>
  </select>
</mapper>

㉟ 웹 서비스 운영 장애 대응 방법 ▼
🚒 소방서 출동 프로토콜로 이해하기
서버 장애는 화재와 같습니다. 침착하게 절차대로 대응해야 합니다.

🚨 1. 탐지 — 모니터링 알람 or 사용자 신고로 화재 감지
📢 2. 알림 — 담당자에게 즉시 연락 (비상 연락망 가동)
💧 3. 초기 진화 — 서버 재시작 or 최근 배포 롤백 (서비스 복구 최우선)
🔍 4. 원인 파악 — 로그 분석, DB 확인, 배포 이력 확인
🔧 5. 근본 해결 — 코드 수정 후 재배포
📋 6. 사후 보고 — 장애 보고서 작성, 재발 방지 대책
⚡ 서비스 복구가 원인 파악보다 먼저다 (롤백 > 디버깅)
📊 CPU, 메모리, 에러율, 응답시간을 실시간 모니터링해야 함
📝 장애 보고서: 발생시각, 영향범위, 원인, 조치, 재발방지
장애 대응 프로세스 (Incident Response)
1. 탐지         모니터링 알람 / 사용자 신고
2. 초기 대응    장애 등급 분류(P1/P2/P3), 비상 연락망 가동
3. 원인 파악    로그 → 에러 시각/스택트레이스, 최근 배포 확인
4. 임시 조치    재시작 / 롤백 (서비스 복구 최우선)
5. RCA 분석     코드버그 / 인프라 / 외부 연동 중 원인 확정
6. 영구 조치    코드 수정 → QA → 배포
7. 사후 보고    장애 보고서, 타임라인, 재발 방지 공유
㊱ 로그 기반 트러블슈팅 방법 ▼
🔍 블랙박스 영상 분석으로 이해하기
사고가 나면 블랙박스 영상을 보고 "언제, 어디서, 무슨 일이 있었는지" 파악하듯, 서버 로그로 문제를 추적합니다.

트러블슈팅 순서:
1. 에러 로그에서 발생 시각과 에러 메시지 확인
2. 에러 직전에 어떤 동작을 했는지 추적
3. 최근 배포가 있었으면 배포 전후 로그 비교
4. DB 슬로우 쿼리 로그도 함께 확인

핵심 명령어: grep "ERROR" app.log — 에러만 뽑아보기
🔎 grep 명령으로 로그에서 에러만 필터링
📡 tail -f 로 실시간 로그 모니터링
🏷️ MDC(요청 추적 ID)로 특정 요청의 전체 흐름 추적 가능
⚠️ OOM, 데드락 등은 스레드 덤프/힙 덤프 분석 필요
Shell — 로그 분석 핵심 명령
tail -f /logs/app.log | grep "ERROR"                        # 실시간
grep "ERROR" app.log | awk '{print $4}' | sort | uniq -c   # 빈도 집계
grep -A 30 "NullPointerException" app.log | head -50         # 스택트레이스
Java — MDC 요청 추적
@Component
public class RequestIdFilter implements Filter {
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) {
        MDC.put("requestId", UUID.randomUUID().toString().substring(0,8));
        try { chain.doFilter(req,res); }
        finally { MDC.clear(); } // 스레드 풀 오염 방지
    }
}
